## 2. "unknown 395"의 정체 — `category` 축이 별도로 필요 (설계 보완점)

`unknown` 상위 링크 항목은 **인식 불가가 아니라, 동물 기원이 아닐 뿐인 정상 성분**이다:

```
로즈마리 추출물(175) · 비타민 E(143) · 타우린(140) · 아마씨(102) · 글리세린(95)
· 고구마(87) · 혼합 토코페롤(73) · 비타민·미네랄 프리믹스(64) · 당근(63)
· 정제수(55) · 소르빈산칼륨(50) · 식물성 오일(45) · 전분(44) · 타피오카 전분(42)
· 미네랄(39) · 해바라기유(37) · 유카 추출물(37) · 보리(37) · 감자(37)
· 카라기난(35) · 귀리(35) · 비트 펄프(32) · 식물성 단백질(32) ...
```

**설계 반영(중요):**
- canonical 모델은 **`source_group`(알러지 기원) 와 `category`(비타민/채소/탄수화물/보존제/오일/첨가물)를 분리**해야 한다. 비타민·채소류는 source_group이 `null`이어도 정상이다(알러지 무관).
- 이들 대부분은 **직접 이름 매칭/별칭**으로 canonical 해석되며, 닭처럼 form 분기가 필요 없다.
- 따라서 통합 설계서 4-1의 `category` 컬럼이 **선택이 아니라 필수**임이 데이터로 확인됨.

**⚠ muscle 버킷의 오분류(검수 필수 증거):**
- `치킨 플레이버` → 실제 **flavor**(향료) 인데 muscle(exact)로 잡힘 → 잘못된 하드 충돌 위험.
- `닭고기 추출물` → **extract**.
- `치킨 파우더` → **meal**(분말).
- `계육 부산물` → **by-product**(부산물, caution).
- `육류 (닭고기, 소고기, 생선 등)` → **복합/다원료** → 분해 불가, 검수 필요.

이들은 드라이런에서 `needs_manual_review=true`로 플래그된다(`name_ko ~ '\(|,|등|...'` 및 장기/육수 조건).

## 5. 병합 시 기술적 위험 — `product_ingredients` PK 충돌

`product_ingredients` PK = `(product_id, ingredient_id)`. 두 표면형이 **같은 제품에 동시 링크**돼 있으면, canonical로 합칠 때 PK 충돌 발생.
→ 병합 SQL은 단순 `UPDATE ingredient_id` 가 아니라 **(a) 원문 스냅샷(`product_ingredient_inputs`) 선저장 → (b) 중복 링크 dedup(`sort_order` 최소값 유지) → (c) canonical 재지정** 순서여야 한다. (Phase 1 06 스냅샷 테이블이 이 선행조건.)
